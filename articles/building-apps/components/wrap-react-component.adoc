= Wrap a React Component
:toclevels: 2

This article shows how to use an existing React component in a Vaadin Flow application. The integration involves two files: a Java class that extends `ReactAdapterComponent` and a TypeScript file that bridges the React component to a web component. For wrapping plain web components, see <<wrap-web-component#,Wrap a Web Component>>. For the full reference, see <</flow/integrations/react#,Using React Components in Flow>>.


== Copy-Paste Example

Two files that wrap the https://omgovich.github.io/react-colorful/[react-colorful] color picker.

*Java component* — `RgbaColorPicker.java`:

[source,java]
----
import com.vaadin.flow.component.Tag;
import com.vaadin.flow.component.dependency.JsModule;
import com.vaadin.flow.component.dependency.NpmPackage;
import com.vaadin.flow.component.react.ReactAdapterComponent;
import java.util.function.Consumer;

@NpmPackage(value = "react-colorful", version = "5.6.1")
@JsModule("./rgba-color-picker.tsx")
@Tag("rgba-color-picker")
public class RgbaColorPicker extends ReactAdapterComponent {

    public record RgbaColor(int r, int g, int b, double a) {}

    public RgbaColorPicker() {
        setColor(new RgbaColor(255, 0, 0, 1.0)); // <1>
    }

    public void setColor(RgbaColor color) {
        setState("color", color);
    }

    public RgbaColor getColor() {
        return getState("color", RgbaColor.class);
    }

    public Registration addColorChangeListener(
            Consumer<RgbaColor> listener) {
        return addStateChangeListener("color",
                RgbaColor.class, listener);
    }
}
----
<1> Always call `setState` in the constructor to support `@PreserveOnRefresh`.

*TypeScript adapter* — `frontend/rgba-color-picker.tsx`:

[source,tsx]
----
import { type ReactAdapterElement, RenderHooks }
    from 'Frontend/generated/flow/ReactAdapter';
import { RgbaColorPicker as ReactColorPicker } from 'react-colorful';

class RgbaColorPickerElement
    extends ReactAdapterElement { // <1>

    render(hooks: RenderHooks): React.ReactElement | null {
        const [color, setColor] =
            hooks.useState<RgbaColor>("color"); // <2>
        return <ReactColorPicker color={color}
            onChange={setColor} />; // <3>
    }
}

customElements.define(
    "rgba-color-picker",
    RgbaColorPickerElement); // <4>
----
<1> Extend `ReactAdapterElement` to create a web component that renders React.
<2> `hooks.useState` creates a synchronized state property that the Java side can read and write.
<3> Pass the state to the React component following its API.
<4> Register the custom element with the same tag name used in `@Tag`.


== How It Works

The integration uses an intermediate web component as a bridge:

----
Java (server)  ←→  Web Component (adapter)  ←→  React (client)
----

. The *Java class* extends `ReactAdapterComponent` and communicates state through named properties on the intermediate web component.
. The *TypeScript adapter* extends `ReactAdapterElement`, which is a web component that renders React. It maps web component properties to React state.
. The *React component* renders in the browser, receiving props from and sending updates to the adapter.

This architecture means the React component doesn't need to know about Vaadin, and the Java code doesn't need to know about React. The adapter bridges the two worlds.


== Server-Side: Java Component

Your Java component extends `ReactAdapterComponent` and uses three methods to synchronize state:

[source,java]
----
// Send state from server to client
setState("color", new RgbaColor(255, 0, 0, 1.0));

// Read current state from client
RgbaColor color = getState("color", RgbaColor.class);

// Listen for state changes from client
addStateChangeListener("color", RgbaColor.class, newColor -> {
    // React to color changes
});
----

Use these methods to build a conventional Java API with getters, setters, and event listeners.

The standard annotations work the same as for <<wrap-web-component#,web component wrappers>>:

- `@Tag` — the custom element name for the adapter web component
- `@NpmPackage` — the npm package containing the React component
- `@JsModule` — the TypeScript adapter file


== Client-Side: React Adapter

The TypeScript adapter extends `ReactAdapterElement` and implements a `render()` method that returns React JSX:

[source,tsx]
----
class MyAdapterElement extends ReactAdapterElement {
    render(hooks: RenderHooks): React.ReactElement | null {
        const [value, setValue] = hooks.useState<string>("value");
        return <MyReactComponent value={value} onChange={setValue} />;
    }
}

customElements.define("my-adapter", MyAdapterElement);
----

The `hooks` parameter provides:

- `hooks.useState<T>(name)` — defines a named state property synchronized with the server. Returns a `[value, setter]` pair like React's `useState`.
- `hooks.useCustomEvent<T>(name, init?)` — returns a callback that dispatches a custom DOM event, useful for actions that don't map to state changes.

Register the element with `customElements.define()` using the same tag name as `@Tag` on the Java side.


== State Synchronization

State flows in both directions through named properties on the adapter web component.

*Server to client*: Call `setState("name", value)` in Java. The adapter's `hooks.useState("name")` picks up the new value and React re-renders.

*Client to server*: When the React component calls the setter from `hooks.useState`, the adapter property updates. Call `getState()` or use `addStateChangeListener()` in Java to read or react to the change.

=== Non-Primitive Types

For complex types, define a Java record (or bean) that mirrors the JavaScript object structure:

[source,java]
----
public record RgbaColor(int r, int g, int b, double a) {}
----

The record's field names must match the JavaScript property names. Vaadin serializes the record to JSON when sending to the client and deserializes it back when reading from the client.

=== Initial State

Always call `setState` in the Java constructor to set the initial state:

[source,java]
----
public RgbaColorPicker() {
    setState("color", new RgbaColor(255, 0, 0, 1.0));
}
----

This ensures the state is initialized from the server, which is required for `@PreserveOnRefresh` to work correctly. Without it, the component may lose its state when the user refreshes the page.


== Making It a Form Field

To use a React-based component with Vaadin's Binder, extend `AbstractSinglePropertyField` instead of `ReactAdapterComponent`:

[source,java]
----
@Tag("rgba-color-picker")
@NpmPackage(value = "react-colorful", version = "5.6.1")
@JsModule("./rgba-color-picker.tsx")
public class RgbaColorPickerField
        extends AbstractSinglePropertyField<RgbaColorPickerField, String> {

    public RgbaColorPickerField() {
        super("color", "", false);
    }
}
----

This gives you `getValue()`, `setValue()`, `addValueChangeListener()`, and full Binder compatibility. The TypeScript adapter remains the same.


== Pitfalls

*Call `setState` in the constructor.* If you skip this, `@PreserveOnRefresh` doesn't work because the server has no initial state to restore. Always set a default value in the constructor.

*Match tag names between Java and TypeScript.* The `@Tag` annotation value and the string in `customElements.define()` must be identical. A mismatch means the web component never connects and nothing renders.

*Keep the adapter thin.* The TypeScript adapter should only map between web component properties and React props. Don't put business logic or complex state management in the adapter — keep that on the server side in Java.

*Use records for complex state.* When sending objects between Java and TypeScript, use Java records with field names that match the JavaScript property names. Vaadin handles JSON serialization automatically, but name mismatches cause silent failures.
