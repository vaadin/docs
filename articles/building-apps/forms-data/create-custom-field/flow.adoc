== Custom Fields in Flow

If you have multiple components that you want to act as a single field, or just a component that does not have support for use with Binder, setting a label, or setting a helper message, you will likely want to wrap those components within a Custom Field.
To create a Custom Field in Flow you will need to create a class extending `CustomField` and specify a value type that will represent the combined value of all its child components.
You will also have to implement two methods `generateModelValue` (to compute the Custom Field value from child components) and `setPresentationValue` (to distribute the Custom Field value to child components). These methods will be called automatically once a value change is detected on client-side or server-side respectively.

== Data Binding

There are two main approaches for interacting and processing the value in your Custom Field.

* Through direct interaction with the Custom Field
* Through Binder

The approach that you should use depends on the context of where and how your field is being used. If it's part of a form with multiple components, you should likely use it with the binder. If it's on its own or it's not relevant to the data object in the binder (for example, if the Custom Field is only used to filter some data in the form) then you should likely interact with it directly. We will cover both cases below.

=== Direct interactions

CustomField implements the `HasValue` interface, which means you get the same API used in many other fields to interact with its data. With `myCustomField.getValue()` you can retrieve the current value and with `myCustomField.setValue(value)` you can update the value.

If you need to be aware of your Custom Field's value changing, then you can listen to the value change event by passing a listener to  `myCustomField.addValueChangeListener(listener)`. It is recommended that you either use a lambda expression (if it will contain only few lines of code) or a method reference in place of a listener. The received event object will contain the new value (through `event.getValue()`), the old value (through `event.getOldValue()`), reference to your custom field (through `event.getSource()`), and whether the change is coming from the client (through `event.isFromClient()`) instead of being set programmatically with `myCustomField.setValue(value)` for example.
```java
myCustomField.addValueChangeListener(event -> {
    if (event.isFromClient()) {
        var value = event.getValue();
        fullNameDiv.setText("Updated name: %s %s".formatted(value.getFirstName(), value.getLastName()));
    }
});
```

=== Interactions through Binder

Just like with the direct interactions there's really no difference between when you are binding a `TextField`, a `ComboBox` or a Custom Field.
You should be already familiar with the Binder, how set up value binding and use features like validation.

Here's an example of binding a Custom Field to an existing `Binder<MyBinderObject> binder` instance:
```java
binder.forField(myCustomField)
    .withValidator(new MyValidator())
    .bind(MyBinderObject::getName, MyBinderObject::setName);
```

In this example, the `MyBinderObject.getName()` (and `setName(..)`) should match the same object type as the Custom Field's value, meaning if your Custom Field extends `CustomField<MyNameObject>` the return type for `MyBinderObject.getName()` should be `MyNameObject`.

Now when you call `binder.setBean(myObject)` or `binder.readBean(myObject)` your Custom Field should receive the relevant value.

== Label

When you implement a CustomField you get a label out of the box, it is recommended that you take advantage of this feature,
as it looks and behaves just like the labels for other Vaadin field components. That makes it easy to maintain the same label
styling between different types of fields.
You can use `customField.setLabel("My field")` to set the label text.

In case you need custom label, for example, if you have multiple input fields and need a label for each of them, you'll
need to use the `NativeLabel` component. You will also have to override the `getLabel()` and `setLabel(String text)` methods so that they interact with the new label. You should also make the `NativeLabel` point to the input field by using the `label.setFor(Component)` or `label.setFor(String)` method. The input field should have a unique id set on it, and that id should be specified when using `label.setFor(String)` method. In cases where there are multiple input fields are present, you can either use individual `NativeLabel` to describe them, or use `aria-labelledby` attribute on each input field pointing to a specific `NativeLabel` id, in which case multiple inputs could point to the same label.

```java
public class MyCustomField extends CustomField<Integer> {
    // ...
    private final NativeLabel label;
    private final Input input;

    public MyCustomField(String label) {
        input = createInput();
        label = new NativeLabel(label);
        label.setFor(input);

        var layout = new Div(label, input);
        add(layout);
    }

    @Override
    public String getLabel() {
        return label.getText();
    }

    @Override
    public void setLabel(String labelText) {
        label.setText(labelText);
    }

    private Input createInput() {
        var input = new Input();
        // Generate a (very likely) unique field id
        input.setId("my-custom-field-" + RandomGenerator.getDefault().nextInt(1, 100_000));
        // or input.setAriaLabelledBy("my-custom-field-label-...");
        return input;
    }
}
```
