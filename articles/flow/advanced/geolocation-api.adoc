= Geolocation API
:toc:

The [classname]`Geolocation` API gives server-side Java code access to the https://developer.mozilla.org/en-US/docs/Web/API/Geolocation_API[browser Geolocation API]. It exposes two static entry points: [methodname]`Geolocation.getPosition()` for a single reading and [methodname]`Geolocation.watchPosition()` for a continuous stream that returns a [classname]`GeolocationWatcher` handle.

[NOTE]
The browser shows its own permission dialog the first time the application asks for a location. The dialog is controlled by the browser -- Vaadin can't style it, suppress it, or detect when it's shown. Geolocation also requires a secure context (HTTPS), except on `localhost` during development.


== Getting the Current Position

Use [methodname]`Geolocation.getPosition()` to request the device's current position once -- the typical pattern for a "Use my location" button. The call takes two consumers, one for the [classname]`GeolocationPosition` and one for the [classname]`GeolocationError`; whichever the browser produces is delivered on the UI thread:

[source,java]
----
Button locate = new Button("Use my location", e -> Geolocation.getPosition(
        pos -> {
            double lat = pos.coords().latitude();
            double lon = pos.coords().longitude();
            map.setCenter(new Coordinate(lon, lat));
        },
        err -> Notification.show("Could not determine your location.")));
----

The call returns immediately. Exactly one of the consumers is invoked once the browser reports a result. See <<#handling-errors,Handling Errors>> for the full set of error reasons; the browser-supplied [methodname]`err.message()` is meant for diagnostic logs, not for the user-facing message.


== Options

Configure a request with [classname]`GeolocationOptions`. Use the builder rather than the canonical constructor -- it accepts [classname]`Duration` for the time-based fields and gives every option a named setter:

[source,java]
----
GeolocationOptions opts = GeolocationOptions.builder()
        .highAccuracy(true)
        .timeout(Duration.ofSeconds(10))
        .maximumAge(Duration.ZERO)
        .build();

Geolocation.getPosition(pos -> { /* ... */ }, err -> { /* ... */ }, opts);
----

`highAccuracy(boolean)`:: Asks for the most accurate reading the device can produce -- typically GPS on mobile devices. Uses more battery and can take longer. Default: `false`.
`timeout(Duration)`:: Maximum time the browser may spend before failing with a [code]`TIMEOUT` error. Default: no timeout.
`maximumAge(Duration)`:: How old a cached reading may be while still acceptable. [code]`Duration.ZERO` (the default) refuses cached readings. Larger values save battery.

A setter that's never called leaves the corresponding field at the browser default. There are also [code]`int` overloads if you already have a millisecond value on hand.


== Tracking Position Changes

[methodname]`Geolocation.watchPosition(Component)` starts a continuous watch and returns a [classname]`GeolocationWatcher` tied to the owner component. The browser reports new positions as the device moves. There are two ways to consume them -- pick the one that fits the destination.

=== Listener Style

Use [methodname]`addPositionListener()` when every reading needs to be processed in order -- for example, persisting each point to a database, appending to a route log, or forwarding to an external service. The position consumer fires for every successful reading and the error consumer fires for every error; the initial pending state is never delivered.

[source,java]
----
GeolocationWatcher watcher = Geolocation.watchPosition(this);
watcher.addPositionListener(
        pos -> repository.save(pos),
        err -> log.warn("location error: {}", err.message()));
----

Both consumers are required. To opt out of one side, pass an empty lambda explicitly. Add multiple listeners if needed; the returned [classname]`Registration` removes both at once. Listeners stay attached across [methodname]`stop()`/[methodname]`resume()` cycles.


=== Signal Style

Use [methodname]`positionSignal()` when the goal is to reflect the *current* position in the UI -- the signal always holds the latest reading, so subscribers naturally show the most recent value and the listener-style "process every point" semantics don't get in the way. The signal value is one of three sealed [classname]`GeolocationResult` subtypes, so a [code]`switch` is exhaustive at compile time:

[source,java]
----
GeolocationWatcher watcher = Geolocation.watchPosition(this);

Signal.effect(this, () -> {
    switch (watcher.positionSignal().get()) {
    case GeolocationPending p ->
        status.setText("Waiting for first reading…");
    case GeolocationPosition pos -> {
        status.setText("");
        map.setCenter(new Coordinate(pos.coords().longitude(),
                pos.coords().latitude()));
    }
    case GeolocationError err ->
        status.setText("Error: " + err.message());
    }
});
----

`GeolocationPending`:: Initial state, held until the browser reports the first reading.
`GeolocationPosition`:: A successful reading. Replaces the previous value on every update.
`GeolocationError`:: The browser reported an error.

The browser watch is cancelled automatically when the owner component detaches, so most applications never need cleanup code. Pass [classname]`GeolocationOptions` as a second argument to control accuracy:

[source,java]
----
GeolocationWatcher watcher = Geolocation.watchPosition(this,
        GeolocationOptions.builder().highAccuracy(true).build());
----


=== Stopping and Resuming

Call [methodname]`stop()` to end tracking from application code -- for instance, behind a "Stop tracking" button. The handle stays usable: [methodname]`resume()` installs a fresh browser watch on the same watcher and any signal subscribers and listeners keep working.

[methodname]`activeSignal()` reports whether the watcher is currently receiving updates. Bind a toggle button to it to avoid tracking a separate flag:

[source,java]
----
Button toggle = new Button();
toggle.addClickListener(e -> {
    if (watcher.activeSignal().peek()) {
        watcher.stop();
    } else {
        watcher.resume();
    }
});

Signal.effect(this, () -> toggle.setText(
        watcher.activeSignal().get() ? "Stop tracking" : "Start tracking"));
----

[methodname]`positionSignal()` is reset to [classname]`GeolocationPending` on each [methodname]`resume()`.

.Permission revoke caveat
[NOTE]
If the user revokes location permission while a watch is active and then grants it again, the browser silently stops delivering updates to the existing watch. To recover, call [methodname]`stop()` followed by [methodname]`resume()`. To do this automatically, subscribe to [methodname]`Geolocation.availabilityHintSignal()` and trigger the cycle when availability returns to [code]`GRANTED`.


[#handling-errors]
== Handling Errors

A [classname]`GeolocationError` carries a numeric [methodname]`code()` and a [methodname]`message()`. Switch on [methodname]`errorCode()` to get a typed [classname]`GeolocationErrorCode` enum -- the [code]`switch` is exhaustive at compile time and unknown future codes from the browser surface as [code]`UNKNOWN` rather than throwing:

[source,java]
----
Geolocation.getPosition(pos -> showOnMap(pos), err -> {
    String userMessage = switch (err.errorCode()) {
        case PERMISSION_DENIED ->
            "Location is blocked. Please enable it in your browser settings.";
        case POSITION_UNAVAILABLE ->
            "Could not determine your location.";
        case TIMEOUT ->
            "Location request took too long. Please try again.";
        case UNKNOWN ->
            err.message();
    };
    Notification.show(userMessage);
});
----

The browser-supplied [methodname]`message()` is meant for logging -- the wording isn't standardized, so don't show it directly to end users.


== Checking Availability

[methodname]`Geolocation.availabilityHintSignal()` is a reactive signal of [classname]`GeolocationAvailability` reporting whether the feature is usable and what permission state the origin has. Reading it does *not* trigger a permission dialog -- it tells you whether one would appear on the next [methodname]`getPosition()` or [methodname]`watchPosition()` call.

[cols="1,3"]
|===
| Value | Meaning

| `GRANTED`
| Permission already granted. A subsequent call proceeds silently.

| `DENIED`
| Permission already denied. Subsequent calls fail with `PERMISSION_DENIED` without prompting.

| `PROMPT`
| Not yet decided. The next call shows the browser's permission dialog.

| `UNKNOWN`
| The browser didn't report a state. Treat as "wait for an explicit user action".

| `UNSUPPORTED`
| The page is served over plain HTTP, or an iframe Permissions-Policy blocks geolocation. No user action can change this.
|===

Typical use is to decide how to render before the user clicks anything -- for example, auto-fetching on return visits when permission is already granted, or hiding the location button entirely when the feature is unsupported:

[source,java]
----
@Override
protected void onAttach(AttachEvent event) {
    if (Geolocation.availabilityHintSignal()
            .peek() == GeolocationAvailability.GRANTED) {
        Geolocation.getPosition(this::populateLocalContent,
                err -> { /* ignore */ });
    }
}
----

Subscribe with [methodname]`Signal.effect()` to react to changes -- for instance, when the user revokes permission while the view is open.

.Reliability caveats
[NOTE]
The value is a hint, not authoritative. Safari always reports [code]`UNKNOWN` because it doesn't expose permission state for geolocation. Firefox doesn't always propagate permission changes the user makes in browser settings. Chromium updates promptly but with a small delay. For critical paths, call [methodname]`getPosition()` and rely on the result in the callback.


== Calling From Background Threads

The static methods above resolve the target UI through [methodname]`UI.getCurrent()`, which only works on a request-handling thread. Background threads (executors, scheduled tasks, push handlers) need to pass the UI explicitly. Every entry point has an overload accepting a [classname]`UI` argument:

[source,java]
----
UI ui = UI.getCurrent();
backgroundExecutor.submit(() -> {
    Geolocation.getPosition(pos -> ui.access(() -> updateMap(pos)),
            err  -> ui.access(() -> showError(err)),
            ui);
});
----

[methodname]`Geolocation.availabilityHintSignal(ui)` and the four-argument [methodname]`getPosition(onSuccess, onError, options, ui)` overload follow the same pattern. [methodname]`watchPosition(Component)` already resolves the UI through the owner component, so it works from any thread without a special overload.


== Position Data

A [classname]`GeolocationPosition` contains [classname]`GeolocationCoordinates` and a [methodname]`timestamp()` (milliseconds since the Unix epoch -- use [methodname]`timestampAsInstant()` for an [classname]`Instant`).

[cols="1,1,2"]
|===
| Field | Type | Description

| `latitude()`
| `double`
| Decimal degrees, positive north. Always present.

| `longitude()`
| `double`
| Decimal degrees, positive east. Always present.

| `accuracy()`
| `double`
| 1-sigma horizontal accuracy in meters -- the true position lies within this distance with about 68% probability. Always present.

| `altitude()`
| `Double`
| Meters above the WGS 84 ellipsoid, or `null` when the device can't measure it.

| `altitudeAccuracy()`
| `Double`
| 1-sigma vertical accuracy in meters, or `null` when altitude isn't available.

| `heading()`
| `Double`
| Direction of travel in degrees clockwise from true north (0 = north, 90 = east), or `null` when the device can't measure it or the user is stationary.

| `speed()`
| `Double`
| Ground speed in meters per second, or `null` when the device can't measure it.
|===

Boxed [code]`Double` fields can be [code]`null`. A laptop without GPS typically reports [code]`null` for altitude, heading, and speed; a mobile phone with GPS typically reports all of them.
